# R-NaD モデルアーキテクチャ ドキュメント

このドキュメントでは、Slay the Spire 2 AI における R-NaD モデルの構造化辞書入力およびエキスパート分岐アーキテクチャについて説明します。

## 1. システム概要

モデルは、単一のフラットなベクトルから、**構造化辞書入力 + タスク固有エキスパート (MoE)** アーキテクチャに移行しました。これにより、戦闘、マップ、イベントの各状態を、より効率的かつ専門的に処理することが可能になります。

### 入力構造 (PyTree)

モデルへの入力（および `rnad_bridge.py` の `encode_state` の出力）は、以下の辞書形式です：

```python
{
    "global": [batch, 32],       # 全状態共通の特徴（ゴールド、HP、階層など）
    "combat": [batch, 128],      # 戦闘固有（手札、敵情報など）
    "map": [batch, 64],          # マップ固有（ノードの配置など）
    "event": [batch, 64],        # イベント/報酬の選択肢
    "state_type": [batch],       # インデックス: 0=戦闘, 1=マップ, 2=イベント
}
```

## 2. モデル構造 (`rnad.py`)

`TransformerNet` は主に 3 つの部分で構成されています：

1.  **Global Backbone**: すべての状態に共通の "global" 特徴を処理する共有線形層。
2.  **Expert ユニット**:
    - **`CombatExpert`**: シーケンス処理のための Transformer ベースのモジュール。
    - **`SimpleExpert` (Map/Event)**: 非戦闘時のロジックのための軽量な MLP モジュール。
3.  **Unified Heads**: エキスパートの出力をアクション（Policy）と状態価値（Value）に変換する共有ヘッド。

### 実行フロー

順伝播（Forward pass）は、`jax.vmap` 内で `jax.lax.switch` を使用します：
- **戦闘状態**は Transformer エキスパートにルーティングされます。
- **マップ/イベント状態**は MLP エキスパートにルーティングされます。
- これにより、非戦闘時の意思決定において、重い Transformer ブロックの計算をスキップし、計算コストを大幅に削減します。

---

## 3. 拡張ガイドライン

### 新しい状態タイプ（例：「ショップ」）を追加する場合

特定の画面（ショップなど）専用のネットワークを追加する場合は、以下の手順に従ってください：

1.  **`rnad_bridge.py` の修正**:
    - `encode_state` を更新し、新しい状態を認識するようにします。
    - 辞書に新しいキー（例: `"shop"`) を追加し、関連するデータを格納します。
    - `state_type` に新しいインデックス（例: `3`）を割り当てます。

2.  **`rnad.py` の修正**:
    - 新しいエキスパートクラス（例: `_ShopExpert`）を作成するか、`_SimpleExpert` を再利用します。
    - `TransformerNet.__init__` でエキスパートを初期化します。
    - `TransformerNet.__call__` の `jax.lax.switch` リストに新しいエキスパートを追加します。
    - **重要**: `if hk.running_init():` ブロック内で新しいエキスパートを一度呼び出し、パラメータが正しく登録されるようにします。

3.  **アクションマッピングの修正**:
    - `rnad_bridge.py` の `predict_action` において、その画面に固有の新しいアクションを処理できるようにデコードロジックを更新します。
